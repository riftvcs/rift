Rift

Rift ist ein neues Source-Control-System mit eigenem Storage-Modell — kein Git-Wrapper. Es modelliert permissioned Views, private Changes, Secrets, Snapshots und Agent-Workspaces als native Primitives.

Produktthese

Git modelliert Versionierung gut, aber Zugriff, Sichtbarkeit, Secrets, Change-Lifecycle und Agent-Parallelität schlecht. Rift baut deshalb nicht auf Git auf, sondern definiert einen eigenen Source Graph.

Das eigentliche Problem

Nicht „Git ist schlecht“, sondern: Git hat für heutige Arbeitsmodelle die falschen Grenzen.

Repo-zentrierte Berechtigungen

Wer Repo-Zugriff hat, sieht praktisch alles. Keine nativen Permissions pro Datei, Change, Branch oder PR.

  • private Security-Fixes sind schwierig
  • öffentliche/private Subpackages erfordern Repo-Splitting
  • Contractor-/Agent-Views sind Workarounds

Secrets als Sonderfall

`.env` und Credentials landen in externen Tools, obwohl sie lokal am Ende wieder Dateien sind. Source Control kennt ihren Lifecycle nicht.

  • keine native Rotation
  • keine Secret-Diffs mit Redaction
  • kein Zugriffsaudit pro Workspace

Agent-Parallelität

Worktrees, Branches, Staging und Dirty Working Trees sind für viele parallele Agents unhandlich.

  • Clones sind teuer
  • Worktrees blockieren sich
  • Konflikte werden spät sichtbar

Präzise Definition

Rift ist eine Source-Control-Schicht, die Changes, Files, Secrets, Reviews und Releases granular modelliert.

Ein-Satz-Definition

Rift ist ein eigenständiges, permission-aware, agent-natives Source-Control-System, das Changes, Files, Secrets, Branches, Reviews und Releases als granulare, versionierte und berechtigte Objekte modelliert. Es nutzt einen eigenen Object Store und Change Graph; Git-Import/Export ist nur eine mögliche Brücke, nicht die Grundlage.

Kern-Use-Cases

Die spannendsten Workflows entstehen dort, wo Git heute organisatorische Workarounds erzwingt.

Public Repo mit privaten PRs

Security-Fixes oder unfertige Features bleiben privat, obwohl das Repository öffentlich ist. Veröffentlichung kann geplant oder embargoed erfolgen.

private branchembargoed releasepublic reveal

Secrets sicher versionieren

`.env.dev`, `.env.staging` und `.env.prod` sind versionierte Secret Objects mit Scope, Audit und Redaction — nicht chaotische lokale Dateien.

typed secretsrotationaudit

Monorepo mit öffentlichen und privaten Packages

Ein Repo, mehrere Views: Open-Source-SDK, privates Backend, interne Tools und Shared Utilities ohne Submodule-Hölle.

Agent-Swarm arbeitet parallel

Jeder Agent bekommt minimalen Dateiausschnitt, scoped Secrets, isolierten Snapshot, Testumgebung und Change-Provenance.

MVP: Native Rift Local Store

Startpunkt: Nicht Git ersetzen, indem man Git wrapped — sondern den kleinsten eigenen Source-Control-Kern bauen.

MVP-Funktionen

  • CLI rift
  • rift init erzeugt .rift/
  • eigener Object Store unter .rift/objects/
  • SQLite-Metadata für Paths, Snapshots, Changes
  • rift status, rift record, rift log
  • später: Views, Secrets, Workspaces, Server-Protokoll
  • Git Import/Export nur als Migration, nicht als Core

Beispiel Struktur

.rift/
  config.toml
  db.sqlite
  objects/
    ab/abcdef...blob
  refs/
  keys/
  tmp/

Roadmap

Vom lokalen Object Store zu einem permissioned Source Graph für Teams und Agents.

Phase 0

Core Model

Objekte, Trees, Changes, Snapshots und lokale Repository-Struktur definieren.

Phase 1

Local Prototype

rift init/status/record/log ohne Git-Abhängigkeit implementieren.

Phase 2

Views, Secrets, Workspaces

Permissioned Views, Secret Objects und Agent-Workspaces als native Primitives ergänzen.

Phase 3

Server & Interop

Sync-Protokoll, Review-UI, Audit und optionaler Git-Import/-Export für Migration.
